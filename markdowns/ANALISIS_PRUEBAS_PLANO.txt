ANÁLISIS DE PRUEBAS DE CARGA, ESTRÉS Y RENDIMIENTO
Aplicación: Guía Turística Virtual
Herramienta: Apache JMeter 5.6.3

================================================================================
RESUMEN EJECUTIVO
================================================================================

Se realizaron pruebas de carga, estrés, potencia y rendimiento sobre la aplicación de Guía Turística Virtual para determinar el número máximo de usuarios conectados simultáneamente. El sistema fue evaluado bajo dos escenarios: carga base (10 usuarios) y estrés (50 usuarios simultáneos).

RESULTADO PRINCIPAL: El sistema puede soportar al menos 50 usuarios simultáneos con una tasa de éxito del 99.38% y tiempo promedio de respuesta de 144 ms.

================================================================================

- Usuarios simultáneos: 10 usuarios
- Tiempo de rampa: 30 segundos

- Objetivo: Evaluar comportamiento bajo carga normal

GRUPO 2: PRUEBAS DE ESTRÉS
- Usuarios simultáneos: 50 usuarios
- Tiempo de rampa: 60 segundos
- Duración: 300 segundos
- Objetivo: Determinar punto de falla y capacidad máxima

ENDPOINTS EVALUADOS:
1. Frontend Streamlit: Página Principal (/) y Explorar Ciudades (/cities)
2. API Supabase: GET Ciudades (/rest/v1/cities) y GET POIs (/rest/v1/points_of_interest)
3. Integración n8n: Webhook de Recomendaciones (/webhook/tourist-guide)

RECOMENDACIONES
================================================================================

OPTIMIZACIONES RECOMENDADAS:

Para Supabase:
- Implementar caché para consultas frecuentes (ciudades, POIs)
- Optimizar índices en la base de datos para reducir tiempos de respuesta
- Considerar uso de connection pooling
- Implementar retry logic con backoff exponencial para manejar errores ocasionales

Para n8n:
- El sistema ya muestra buen comportamiento, pero se puede optimizar:
- Implementar caché para recomendaciones similares
- Monitorear uso de recursos del servidor n8n bajo carga
- Considerar escalamiento horizontal si se requiere más capacidad

Para el Sistema Completo:
- Implementar load balancing si se requiere escalar más allá de 100 usuarios
- Considerar CDN para assets estáticos del frontend
- Implementar monitoreo en tiempo real de métricas de rendimiento
- Establecer alertas para tasas de error > 2%

PRUEBAS ADICIONALES RECOMENDADAS:

1. Pruebas de Resistencia (Endurance): Ejecutar pruebas durante períodos prolongados (1-2 horas) para identificar posibles memory leaks o degradación gradual.

2. Pruebas de Spike: Evaluar comportamiento ante aumentos súbitos de carga y simular picos de tráfico realistas.

3. Pruebas de Volumen: Incrementar gradualmente hasta encontrar el punto de falla real. Probar con 75, 100, 150 usuarios simultáneos.

4. Pruebas de Escalabilidad Horizontal: Evaluar comportamiento con múltiples instancias y determinar beneficios de arquitectura distribuida.
